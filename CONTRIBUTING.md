# Contributing

This project generates a fully static, Wiktionary-style Interslavic dictionary
(**Slovowiki**) from evidence caches — no server, no database, deterministic
output. There are three ways to contribute: **curate data** (no Rust needed),
**improve the engine** (benchmark-gated), or **improve the site/API**.

## Local workflow

```bash
cargo test                                    # full unit suite, incl. self-verification
cargo run --release -- export --out site      # regenerate the whole site + api/
cd site && python3 -m http.server 8765
```

Then open <http://localhost:8765>.

## Curating data (no code)

### Entry notes — `data/curation-notes.json`

Every generated entry has a **Popraviti / issue** tab that opens a pre-filled
GitHub issue. For local curated notes, create `data/curation-notes.json`
(format in `data/curation-notes.example.json`):

```json
{
  "voda": "Checked manually: matches official form and broad Slavic evidence.",
  "123": "Entry-id notes are supported, but headword keys are more stable."
}
```

Headword keys are folded to the site's standard spelling. Notes appear on the
entry's talk page and on the *Predloženja* (novel-word proposals) page.

### False-friend warnings — `data/semantic-notes.json`

Curated semantic traps applied by `check-text` (CLI **and** the site's
`text-check.html`). Every note must be **anchored to the official gloss** —
state what the dictionary says the word means, then the trap:

```json
{
  "urok": {
    "warning": "Official meaning: 'charm, spell'. Russian speakers may read it as 'lesson' (ru урок) — it is not a lesson in Interslavic.",
    "prefer": ["lekcija"]
  }
}
```

`prefer` entries should themselves be official lemmas. Keys are folded like
curation-note keys.

### Reviewing novel-word proposals

`data/novel-words.tsv` is regenerated by every export but is currently
header-only. Issue #89 found that the official-row pipeline calibrator was
being applied to the corpus path's semantically different coverage score.
Proposal probabilities and buckets now fail closed until a corpus-specific
calibrator is fitted and holdout-validated. Do not restore the worklist by
using raw coverage thresholds or the pipeline calibrator; a future corpus
calibrator must declare and validate its score domain.

## Improving the engine (benchmark-gated)

The core principle: **no algorithmic change is kept unless it improves
measured accuracy** on the reproducible benchmark. The house methodology,
enforced by CI and the report tooling:

1. Gate every consensus rule behind a `ConsensusConfig` flag and add it as a
   rung on the ablation ladder in `eval.rs` (the ladder must end at
   `ConsensusConfig::production()` — a test asserts this).
2. Run `cargo run --release -- evaluate` before and after. The primary
   keep-metric is **exact top-1**; `target/eval/methodology.md` reports each
   rung's **dev/holdout split** (a rule must hold its gain on the held-out
   quarter) and a **paired sign test** (p ≳ 0.05 ⇒ provisional, not proven).
3. Ship a regression test per fix. Record rejected experiments honestly, with
   their measured negative deltas (in-code comment + report).
4. CI fails if exact top-1 drops below the floor in
   `.github/workflows/ci.yml`.

Diagnostic instruments to aim your effort: `audit` (per-stage blame),
`oracle` (per-stage headroom ceilings — reads the answer, diagnostic only),
`rep-eval` / `select-eval` (representative/cluster probes),
`evidence-eval` (the root-absent ceiling), `explain <word>` (full rule trace
for one entry).

## Benchmarks & evaluation commands

| Command | What it measures |
|---|---|
| `evaluate` | the headline benchmark: ablation ladder, holdout split, significance, calibration |
| `corpus-eval` | the site's cognate-set generation path |
| `proto-eval` | the Proto-Slavic rule engine in isolation |
| `derive-eval` | the word-formation layer vs a naive baseline |
| `multiword-eval` | reflexive `X sę`, two-token collocations, historical aspect baseline |
| `aspect-eval` | dedicated ipf↔pf ladder: both/either/pairing correctness, holdout, significance |
| `evidence-eval` | root-absent recoverability + augmentation A/B |
| `inflect-eval` | inflection census + RULE_SPEC §3 grammar invariants |
| `synonym-eval` | synonym-inclusive accuracy (thesaurus-based) |
| `audit` / `oracle` | miss attribution / per-stage ceilings (diagnostic) |
| `check-text <file> [--json]` | verify an Interslavic text against the lexicon |

Every new capability ships with its own benchmark command and a committed
report under `target/eval/` — state the denominator, the leakage story, and
dev/holdout numbers in the report's first paragraph.

## The lexical API and its invariants

`export` writes a static agent-facing API under `site/api/` (see
`api/agent-guide.md` and the README section "Lexical verification API").
Contributor-relevant invariants, all CI-tested or verified:

- **Single source**: HTML inflection tables and `api/forms/<n>.json` records
  are generated from the same cell getters in `src/forms.rs` — never bypass
  them (round-trip tests will catch drift).
- **Wire format is frozen**: the fold table, the FNV-1a/2048 shard router and
  the record array layout are mirrored in client JS and documented in the
  agent guide. Changing any of them is a schema break: bump
  `forms::SCHEMA_VERSION`.
- **Determinism**: no timestamps, `BTreeMap` ordering — two consecutive
  exports must be byte-identical under `api/`.
- **Trust model**: generated (unmatched) corpus lemmas get lemma records with
  null probability until a compatible corpus calibrator exists, and **no
  inflection records**; generated derivatives retain their separate
  pattern-specific holdout lower bound. Derivation proposals are never in the
  verification index.

## Static wiki pages

The exporter also writes generated navigation/tool pages:

- `forms.html` (reverse form lookup) and `text-check.html` (client-side text
  verification)
- `proposals.html` (novel-word proposals), `metrics.html` (accuracy stats)
- `datasets.html` (all machine-readable artifacts), `api/` (agent API)
- `all-pages.html`, `categories.html` + `category/*.html`
- `indices.html` + `index/*.html`, `portals.html` + `portal/*.html`
- `graph.html` / `graph.json`, `what-links-here/*.html`
- `sitemap.xml`, `build.json`

## Useful make targets

- `make export` — regenerate `site/`
- `make serve` — regenerate and serve locally
- `make test` / `cargo test` — run tests
- `make eval` — official dictionary benchmark
- `make corpus-eval` — evaluate the corpus-driven site path
